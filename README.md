# MedCheck - Сайт медицинской диагностики

## Описание проекта

MedCheck - это веб-приложение для медицинской диагностической компании, реализованное с использованием Django и Bootstrap. Проект предоставляет полный функционал для управления медицинскими услугами, записями пациентов и администрированием контента.

## Цветовая схема

- Основной цвет (бирюзовый): внушает доверие и спокойствие,
- Вторичный цвет (коралловый): добавляет теплоту и мотивацию к действию,
- Фоновый цвет (белый): подчёркивает чистоту и порядок.

Это не просто красиво — это работает на конверсию и лояльность пациентов.
Разработано для компании медицинской диагностики с акцентом на современный, профессиональный и дружелюбный интерфейс.

## Функционал сайта

### Основные страницы:
- **Главная страница**: Описание компании, перечень категорий услуг, контактная информация
- **О компании**: История компании, миссия и ценности, команда врачей
- **Услуги**: Перечень и подробное описание медицинских услуг с ценами
- **Контакты**: Адрес, карта проезда, телефоны, email, форма обратной связи
- **Личный кабинет**: Регистрация, авторизация, запись на прием, просмотр истории
- **Кабинет врача**: Просмотр и редактирование записей пациентов и формирование результатов обследования

### Админка Django
- Управление пользователями
- Управление услугами
- Управление записями на прием
- Управление результатами обследования
- Управление контентом сайта

## Установка и запуск

1. Клонировать репозиторий:
```bash
git clone https://github.com/alekseyzvyagintsev/MedCheck.git && cd MedCheck
```

2. Установить зависимости:
```bash
poetry install
```

3. Активировать виртуальное окружение:
```bash
poetry shell
```

После выполнения этой команды вы окажетесь внутри виртуального окружения Poetry, где будут доступны все установленные зависимости. Чтобы выйти из виртуального окружения, используйте команду `exit` или `deactivate`. Для проверки, что вы находитесь внутри виртуального окружения, можно выполнить команду `which python`, которая покажет путь к интерпретатору Python внутри виртуального окружения.

4. Настроить переменные окружения (создать файл .env в корне проекта):
```.env
DEBUG=True
SECRET_KEY=your-secret-key-here
DB_NAME=medcheck
DB_USER=your-db-user
DB_PASSWORD=your-db-password
DB_HOST=localhost
DB_PORT=5432
```

5. Применить миграции:
```bash
python manage.py migrate
```

6. Заполнить базу данных контентом:
```bash
python manage.py fill_db_content
```
Эта команда создает суперпользователя (username: admin, password: admin) и заполняет контент сайта (название, описание, историю компании, и три главные фигуры: Главный врач, старшая медсестра, управляющий отделением).

7. При необходимости заполнить тестовыми данными (сотрудники, категории услуг, услуги):
```bash
python manage.py fill_db
```

8. Запустить сервер:
```bash
python manage.py runserver
```

Проект будет доступен по адресу http://127.0.0.1:8000
Админка доступна по адресу: http://localhost:8000/admin

# Работа с контейнерами
## Контейнеризация (Docker)

Проект полностью контейнеризирован и готов к запуску с помощью Docker.

### Структура контейнеров

Проект использует два контейнера:
- **web**: Веб-приложение Django
- **db**: База данных PostgreSQL

### Запуск контейнеров

1. Создать файл `.env` в корне проекта с переменными окружения:
```env
DEBUG=True
SECRET_KEY=your-secret-key-here
DB_NAME=medcheck
DB_USER=your-db-user
DB_PASSWORD=your-db-password
DB_HOST=db
DB_PORT=5432
```

2. Собрать и запустить контейнеры:
```bash
docker-compose up --build
```
Проект будет доступен по адресу http://127.0.0.1:8000
Админка доступна по адресу: http://localhost:8000/admin

### Особенности контейнерной версии

**Контейнерная версия проекта является демонстрационной моделью со следующими особенностями:**
- Автоматическое создание суперпользователя командой `fill_db_content`, заполнение контент сайта и тестовыми данными
- Структура контента привязана к ID экземпляров моделей `Fragment`, что обеспечивает стабильность отображения
- Запрещено создавать произвольные экземпляры контента, так как это может нарушить структуру сайта
- Все изменения контента рекомендуется вносить через панель администратора после первоначального заполнения командой `fill_db_content`

# Подробности о внутренней логике работы сайта

Сайт MedCheck построен на архитектуре Django с использованием следующих ключевых компонентов:

1. **Модель контента (Fragment)**:
   - Все описательные элементы сайта (заголовки, описания, тексты) хранятся в модели `Fragment`
   - Каждый фрагмент имеет уникальный ID, к которому привязаны шаблоны
   - Это обеспечивает стабильность контента и предотвращает поломку структуры при изменении данных
   - Контент передается на все страницы через функцию `get_common_context()` в `content/content.py`, которая формирует словарь фрагментов по их ID

2. **Модели услуг и категорий (Service и ServiceCategory)**:
   - `ServiceCategory` - категории медицинских услуг с полями: название, описание, порядок отображения
   - `Service` - медицинские услуги с полями: название, категория, описание, цена, длительность, подготовка, время получения результатов
   - Услуги привязаны к врачам через ManyToMany поле `doctors`
   - Цена автоматически форматируется с разделением тысяч через property `price_formatted`

3. **Модель пользователей (User)**:
   - Наследуется от `AbstractUser` с дополнительными полями: email (уникальный), телефон, дата рождения, адрес, фото
   - При регистрации нового пользователя с ролью врача присваивается (is_staff=True), автоматически вызываются методы:
     - `assign_doctor_group()` - назначает пользователя в группу "доктор"
     - `create_doctor_schedule()` - создает график работы на 3 недели вперед
   - Метод `save()` переопределен для автоматической обработки персонала

4. **Модель записи на приём (Appointment)**:
   - Связывает пациента, услугу, врача и дату/время приёма
   - Имеет методы класса для управления записями:
     - `is_slot_available()` - проверяет доступность временного слота
     - `get_available_slots()` - возвращает список доступных слотов для записи
   - Временные слоты привязаны к графику работы врача
   - Используется для отображения записей в профиле пациента и кабинете врача
   - Позволяет пациенту просматривать и отменять свои записи
   - Является основой для создания результатов диагностики

5. **Модель графика работы врача (DoctorSchedule)**:
   - Определяет доступные временные слоты для записи
   - Имеет поля: врач, день недели, время дня (кратностью 30 минут)
   - Использует property `start_datetime` и `end_datetime` для получения полных datetime объектов
   - Создается автоматически при создании врача через `create_doctor_schedule()`

6. **Модель результатов диагностики (DiagnosisResult)**:
   - Связана с записью на приём и содержит файл с результатами
   - Имеет статусы: "в обработке", "готово", "ошибка"
   - Предоставляет связь моделей `appointment` и diagnosis_result через поле `patient` для доступа к пациенту
   - Метод `get_file_url()` возвращает URL файла, а `is_ready()` проверяет готовность результатов
   - Связана с записью на приём, что обеспечивает целостность данных
   - Является основой для системы обработки результатов врачом и их просмотра пациентом

7. **Система заполнения базы данных**:
   - `fill_db_content` - заполняет основной контент сайта и создает суперпользователя
   - `fill_db` - заполняет тестовыми данными (врачи, услуги, категории)

8. **Бизнес-логика приложения**:
   - Реализована в модуле `services/utils/`

   **Модуль appointments.py - пошаговая система записи на приём**:
   - Система записи реализована как пошаговый процесс, разбитый на 5 этапов
   - Каждый шаг представлен отдельным шаблоном в `services/templates/services/steps/`
   - Маршруты определены в `services/urls.py` и последовательно обрабатывают процесс записи
   - Этапы процесса:
     1. `get_services` - выбор услуги из списка доступных
     2. `choose_service` - переход к выбору врача для выбранной услуги
     3. `choose_doctor` - выбор конкретного врача
     4. `choose_date` - выбор даты приёма
     5. `confirm_appointment` - подтверждение записи
   - Для каждого шага существует отдельный шаблон:
     - `step1_choose_service.html` - выбор услуги
     - `step2_choose_doctor.html` - выбор врача
     - `step3_choose_date.html` - выбор даты
     - `step4_choose_time.html` - выбор времени из доступных слотов
     - `step5_confirmation.html` - подтверждение записи
   - Использование пошагового подхода позволяет:
     - Поэтапно собирать необходимые данные
     - Упростить валидацию на каждом этапе
     - Улучшить пользовательский опыт
     - Обеспечить логическую последовательность процесса записи
   - Функция `get_available_slots()` возвращает доступные временные слоты, основываясь на графике работы врача и уже существующих записях
   - Функция `create_appointment_logic()` содержит основную бизнес-логику создания записи с проверкой доступности слота и валидацией времени

   **Модуль doctors.py - управление врачами и их расписанием**:
   - Воспользоваться данной логикой можно через только через профиль сотрудника. Любой сотрудник может зарегистрировать нового врача.
   - `assign_doctor_group(user)` - автоматически назначает нового врача в группу "доктор"
   - `create_doctor_schedule(doctor)` - создает график работы врача на 3 недели вперед с интервалом 30 минут (с 9:00 до 17:00, пн-пт)
   - `get_time_of_day(hour, minute)` - вспомогательная функция для преобразования времени в соответствующий индекс временного слота
   - График работы автоматически создается при регистрации нового врача через переопределенный метод `save()` в модели User
   - Система использует предопределенные временные слоты, что обеспечивает консистентность расписания и упрощает процесс записи

9. **Логика обработки результатов врачом в кабинете врача**:
   - Доступна по адресу `/services/dashboard/` только для пользователей с правами персонала (is_staff=True)
   - На главной странице кабинета отображаются записи на приём на сегодняшний день
   - Для каждой записи доступны следующие действия:
     - Создать результат диагностики (если еще не создан)
     - Просмотреть существующий результат
     - Редактировать результат
     - Удалить результат
   - Создание результата:
     - Осуществляется через форму с полями: файл с результатами, описание, статус
     - Результат привязывается к конкретной записи на приём
     - Статус результата может быть: "в обработке", "готово", "ошибка"
   - Просмотр результата:
     - Отображается полная информация о записи и результатах
     - Пациент, врач и администратор могут просматривать результаты
     - Доступ к результату имеет только врач, который его создал, или суперпользователь
   - Редактирование результата:
     - Доступно только врачу, который создал результат, или суперпользователю
     - Позволяет обновить файл, описание и статус результата
   - Удаление результата:
     - Доступно только врачу, который создал результат, или суперпользователю
     - Реализовано с подтверждением действия через JavaScript, но он не в виде отдельного .js файла, а встроен непосредственно в HTML-шаблоны в виде кнопок подтверждения.
     - После удаления происходит перенаправление в кабинет врача
   - Шаблоны кабинета врача:
     - `dashboard.html` - главная страница кабинета с записями на сегодня
     - `result_form.html` - форма создания/редактирования результатов
     - `result_detail.html` - страница просмотра результатов

10. **Логика просмотра результатов и записей пациентом в личном профиле**:
   - Доступна по адресу `/accounts/profile/` для авторизованных пользователей
   - Страница профиля разделена на секции с якорями:
     - Профиль
     - Записи на прием
     - Результаты диагностики
     - Настройки (профиля пока не реализованы представлены только в качестве демонстрации направления развития)
   - Просмотр записей на приём:
     - Отображаются только активные записи на будущие даты
     - Для каждой записи показываются: услуга, врач, дата и время
     - Пациент может отменить запись через форму с подтверждением
   - Просмотр результатов диагностики:
     - Отображается таблица со всеми результатами пациента
     - Для каждого результата показываются: время приема, услуга, врач, статус
     - Статус отображается цветным бейджем (зеленый - готово, желтый - в обработке, красный - ошибка)
     - При статусе "готово" доступны кнопки "Просмотреть" и "Скачать PDF"
     - При других статусах кнопка "Просмотреть" недоступна
   - Контекст профиля формируется в `accounts/views.py` в функции `profile()`:
     - `active_appointments` - активные записи пациента
     - `diagnosis_results` - результаты диагностики пациента
   - Результаты диагностики упорядочены по дате загрузки (сначала новые)

11. **Административная панель**:
   - Доступна по адресу `/admin`
   - Учетные данные суперпользователя: username: admin, password: admin
   - После заполнения контентом через команды, можно редактировать данные в админке

10. **Работа с контентом**:
   - Все фрагменты контента создаются/обновляются по ID
   - Это гарантирует, что определенный контент всегда будет отображаться в заданном месте сайта
   - Произвольное создание экземпляров `Fragment` не рекомендуется, так как может привести к некорректному отображению
